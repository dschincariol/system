## Canonical Configuration Surfaces

| Surface | Canonical For | Notes |
| --- | --- | --- |
| `.env.example` | Baseline catalog of operator-edited environment variables | Starting point for local `.env` files and service-environment templates. |
| `.env` or service environment | Bootstrap and runtime configuration | Used by `start_system.py`, `dashboard_server.py`, `start_ingestion.py`, and many subsystem modules. |
| `data_sources` table | Provider credentials and source-specific settings | Managed through [ui/data_sources.html](../ui/data_sources.html) and [services/data_source_manager.py](../services/data_source_manager.py). |
| `DATA_SOURCE_MASTER_KEY` or `DATA_SOURCE_MASTER_KEY_FILE` | Encryption root for stored provider credentials | Required for production-like deployments; never stored in the database. Production/live requires canonical base64 text for exactly 32 random bytes; use `openssl rand -base64 32` and prefer a `0600` key file. |
| Runtime-set environment variables | Process identity and resolved paths | Variables such as `TRADING_LOGS`, `TRADING_DATA`, `ENGINE_JOB_NAME`, and `ENGINE_LAUNCHED_BY_SUPERVISOR` are set by the entrypoints and should not be treated as primary operator inputs. |

## Ownership Rules

- Provider credentials should be managed through the data-source control plane, not as long-lived `.env` values.
- `.env` remains canonical for bootstrap settings such as bind host, port, DB path, execution mode, feature toggles, and master-key discovery.
- `engine/runtime/config_schema.py` validates only part of the full environment contract today. Other variables are still consumed directly by entrypoints and subsystem modules.

### Live Ingestion Writer Tuning Inventory

The bounded parser in [engine/runtime/ingestion_tuning.py](../engine/runtime/ingestion_tuning.py) is the production source of truth for these live ingestion writer knobs. Explicit out-of-bound values are reported as preflight/startup errors rather than silently becoming larger loss windows.

| Area | Knobs | Safe default | `host_32t_123g` profile | Production enforcement |
| --- | --- | ---: | ---: | --- |
| Runtime Postgres pool | `TS_PG_POOL_MIN_SIZE`, `TS_PG_POOL_SIZE`, `TS_PG_POOL_PROFILE` | app `4`, ingestion `8`, jobs `2` | app `4`, ingestion `12`, jobs `3` | `storage_pool.default_pool_size()` by role; `start_ingestion.py` sets ingestion role, child jobs get explicit smaller pools. |
| Ingestion child pools | `INGESTION_CHILD_TS_PG_POOL_*`, `INGESTION_CHILD_TIMESCALE_POOL_MAX_SIZE`, `INGESTION_CHILD_TIMESCALE_PRICES_POOL_MAX_SIZE` | PG `2`, Timescale `2`, prices `2` | PG `3`, Timescale `4`, prices `4` | `ingestion_runtime._spawn_child_once()` overwrites child env unless `INGESTION_CHILD_INHERIT_TS_PG_POOL_PROFILE=1`. |
| Redis hot path | `TS_REDIS_POOL_SIZE`, `TS_REDIS_CONNECT_TIMEOUT_S`, `TS_REDIS_SOCKET_TIMEOUT_S` | `16`, `0.25s`, `0.25s` | `32`, `0.25s`, `0.25s` | `engine/cache/redis_pool.py` clamps pool and timeout values. |
| Timescale telemetry writer | `TIMESCALE_POOL_*`, `TIMESCALE_BATCH_SIZE`, `TIMESCALE_QUEUE_MAXSIZE`, `TIMESCALE_FLUSH_INTERVAL_S`, retry/backpressure/timeouts | pool `4`, batch `500`, queue `1024`, flush `1.0s` | pool `8`, batch `1000`, queue `512`, flush `0.5s` | `TimescaleConfig.from_env()` clamps values; enqueue backpressure raises `TimescaleBackpressureError`. |
| Price storage sidecar | `TIMESCALE_PRICES_POOL_*`, connect/lock/command/idle timeouts, retry knobs | pool `4` | pool `8` | `PostgresPriceStorageConfig.from_env()` clamps values and records write duration/retry/drop counters. |
| Async price writer | `ASYNC_PRICE_WRITER_QUEUE_MAXSIZE`, `ASYNC_PRICE_WRITER_BATCH_SIZE`, `ASYNC_PRICE_WRITER_FLUSH_INTERVAL_S`, `ASYNC_PRICE_WRITER_HIGH_WATERMARK_RATIO`, `ASYNC_PRICE_WRITER_SHUTDOWN_DRAIN_MAX_S`, `ASYNC_PRICE_WRITER_SPOOL_PATH`, `ASYNC_PRICE_WRITER_SPOOL_MAX_BYTES`, `ASYNC_PRICE_WRITER_SPOOL_BUSY_TIMEOUT_MS`, `ASYNC_PRICE_WRITER_SPOOL_SYNCHRONOUS`, retry/enqueue/dead-letter knobs | queue `2048`, batch `256`, flush `0.5s`, watermark `0.75`, drain cap `30s`, spool bytes `268435456`, sync `FULL` | queue `1024`, batch `512`, flush `0.25s`, watermark `0.75`, drain cap `30s`, spool bytes `268435456`, sync `FULL` | Enqueue persists rows to the bounded SQLite WAL spool before returning, reports high-watermark/backpressure before overflow, and replays spooled rows on startup. Shutdown uses an adaptive drain bounded by the hard drain cap; rows still pending at the deadline remain in the durable spool and are counted as `residual_spooled_rows`. |
| Telemetry append buffer | `TELEMETRY_APPEND_BUFFER_*` | batch `128`, rows `4096`, flush `0.5s` | batch `256`, rows `4096`, flush `0.25s` | `telemetry_append_buffer.py` clamps values and reports queue depth, flush latency, DB duration, retries, drops. |
| Ancillary ingestion observability buffers | `EVENT_LOG_BUFFER_*`, `RUNTIME_METRICS_*`, `RUNTIME_META_BEST_EFFORT_*` | event rows `2048`, metric rows `4096`, meta keys `512` | unchanged | Event log, metrics store, and runtime meta use the same bounded parser; their capacity is included in the buffered-row risk estimate. |

## Strictly Validated Runtime Variables

| Variable | Meaning |
| --- | --- |
| `ENV` | Canonical environment identity used by `load_runtime_config()`. Accepts `dev`, `prod`, or `test`. |
| `RUNTIME_WORKLOAD_PROFILE` | Runtime workload identity. Use `live` for ingestion/scoring/execution runtimes and `offline` only for isolated research/training workers. |
| `PROD_LOCK` | Prevents unsafe production behavior such as enabling training in a locked production runtime. |
| `ALLOW_TRAINING` | Allows or forbids training work in the current runtime. |
| `OFFLINE_TRAINING_LIVE_PROFILE_ACK`, `OFFLINE_TRAINING_LIVE_PROFILE_OWNER`, `OFFLINE_TRAINING_LIVE_PROFILE_REASON` | Explicit acknowledgement contract for offline/heavy work in a live profile. Strict preflight rejects live-profile `ALLOW_TRAINING`, auto-pipeline, drift retrain, tuning, TSFresh live compute, TS foundation, graph relational, LLM discovery, or SHAP live compute settings unless the ack phrase is exactly `I_UNDERSTAND_OFFLINE_TRAINING_IN_LIVE_PROFILE` and owner/reason are non-placeholder. Job launch and direct training entrypoints also block offline jobs in a live profile without this acknowledgement. |
| `TSFRESH_SNAPSHOT_SYMBOL_LIMIT`, `TSFRESH_SNAPSHOT_MAX_SYMBOLS`, `TSFRESH_SNAPSHOT_BATCH_SIZE`, `TSFRESH_SNAPSHOT_MAX_BATCH_SIZE` | Bounded TSFresh materialization workset and write batch controls. Live defaults are intentionally small; offline profile raises them for isolated snapshot jobs. |
| `TUNE_N_TRIALS`, `TUNE_MAX_N_TRIALS` | Optuna tuning trial budget and upper bound. Live profile defaults are small and direct tuning still requires offline profile or live-profile acknowledgement. |

| `EXEC_DEGRADE_BLOCK`, `EXEC_DEGRADE_WARN_COST_PCT`, `EXEC_DEGRADE_CRIT_COST_PCT` | Execution-degradation thresholds surfaced through system and risk APIs. |

## Operational Rules

- In live-capital-sensitive workflows, prefer explicit safe defaults and fail-closed behavior.
- If a provider credential exists in both `.env` and the `data_sources` table, the DB-backed control plane is the intended long-lived source of truth.
- When adding a new operator-owned configuration variable, update `.env.example`, this glossary, and any relevant subsystem README in the same change.

## CRED-UX-1 — Restore the dead data-source credential-entry actions ("Test & Save", "Populate Now") (**Medium**)

**Context.** The Data Sources control-plane page is the operator's primary, non-technical path for entering provider credentials and verifying them (Edit → paste key → Save → Test Connection). Two of its actions are advertised in the UI but dead at runtime, so a non-technical operator following the documented workflow hits failures: `POST /api/data_sources/test_save` ("Test & Save") and `POST /api/data_sources/populate_now` ("Populate Now").

**Defect (confirmed).** Both endpoints fail at runtime despite valid specs and importable, real handlers: `POST /api/data_sources/test_save` returns `{"error":"unknown_endpoint"}`; `POST /api/data_sources/populate_now` returns a stdlib 404. The handlers exist and import cleanly — `routes/data_sources_routes.py:436` `api_post_data_source_populate_now` (bounded one-shot provider populate + storage-contract check) and `routes/data_sources_routes.py:452` `api_post_data_source_test_save` (test-then-persist) — and their route specs are declared at `routes/data_sources_routes.py:29-30`.

**Root cause (confirmed).** Same silent-drop family as UIA-6. `dashboard_server.py` imports only `create/update/delete/enable/disable/account_update/test` from `routes.data_sources_routes` (`dashboard_server.py:5677-5697`, with `None` fallbacks in the `except`) and registers only those names in the literal `API_HANDLERS` dict (`dashboard_server.py:5790-5796`). `api_post_data_source_test_save` and `api_post_data_source_populate_now` are **absent from both** (verified: 0 references in `dashboard_server.py`). At `dashboard_server.py:5912`, `ROUTE_SPECS = _filter_route_specs_for_handlers(ROUTE_SPECS, API_HANDLERS)` silently drops any spec whose handler name is missing from `API_HANDLERS`, so neither route ever binds. Unmatched GET → stdlib static-404; unmatched non-GET → `unknown_endpoint` (`engine/api/http_transport.py`).

**Your task.** Implement the optimal fix so the credential-entry UX works end to end. (1) **Register both routes:** add `api_post_data_source_test_save` and `api_post_data_source_populate_now` to the import block (and their `None` fallbacks in the `except`), and add both entries to `API_HANDLERS`, so they resolve to the real handlers. (2) **Verify the full operator flow** — Save (`/update`), Test (`/test`), Test & Save (`/test_save`), and Populate Now (`/populate_now`) — is reachable and functional from the Data Sources page, and that `account_update` provider-account credential entry is also intact. (3) **Systemic guard (coordinate with UIA-6, do not duplicate):** ensure the boot-time fail-loud check for silently-dropped advertised routes covers these two; if the UIA-6 guard already exists, these routes must now pass it; if it does not yet exist, add the minimal startup assertion/diagnostic so that any `ROUTE_SPECS` entry whose handler is missing from `API_HANDLERS` fails loudly at boot (and in `tools/validate_repo.py` / the UI contract checker) instead of being silently filtered. Decide the right home so the check runs on every boot.

**Falsify.** Verify both endpoints return their real payloads (not 404 / `unknown_endpoint`), and that the fail-loud guard actually fires when a handler entry is removed (add a temporary missing entry in a test to prove it trips). Note: `POST /api/data_sources/test_save` persists a source and triggers a provider call, and `POST /api/data_sources/populate_now` runs a bounded one-shot populate — exercise them only with clearly-fake test data and reverse the mutation, or assert via the contract checker rather than a live mutation.

## FEED-TRUTH-1 — Stop the dashboard reporting simulated / no-credential feeds as live-green (**High**)

**Context.** Operators rely on the dashboard's data-source connection tests and ingestion/feed-health surfaces to know whether *real* market data and feeds are flowing before trusting or going live. In the safe/Test profile the system intentionally runs without real credentials and substitutes a simulated price provider — but that substitution must never be presented to an operator as a healthy live feed.

**Defect (confirmed).** With no real credentials configured (the paid-feed secret stubs `polygon_api_key`, `alpaca_key_id`/`alpaca_secret_key`, `tradier_api_token`, `openai_api_key` are present but empty), the dashboard health/test surfaces can report market data as healthy/green because the connection test and the ingestion-health aggregate evaluate the **simulated** provider as a pass — even though the catalog explicitly marks it "This is not a production provider and must not be treated as live market-data success" (`services/data_source_manager.py:1074`). This is "fake-green": a green light that does not mean real data is arriving.

**Root cause (to confirm).** (a) `safe_no_credential_market_data_mode()` (`services/data_source_manager.py:216-229`) plus `apply_safe_no_credential_runtime_environment()` (`:255-262`) strip every credential env var and apply `_SAFE_NO_CREDENTIAL_ENV` (`:140-149`, including `SIMULATED_MARKET_DATA_ENABLED=1`), forcing the simulated provider when `ENV=dev|test`, `ENGINE_MODE=safe`, sim broker, kill-switch on. (b) Health already carries a per-source/per-provider `simulated` boolean (`engine/runtime/health.py:2932,3403,3439,3481,3509`), but the aggregate ingestion health counts a pipeline as healthy whenever `ok and not stale` regardless of whether it is simulated-only (`engine/runtime/ingestion_status.py:261-275`), and the simulated connection test returns `pass` without distinguishing it from a live feed. (c) Net: the top-line health/test status can read green while every credentialed feed is unconfigured. Confirm by reproducing the safe/Test profile with empty secret stubs and reading the health/test payloads.

**Your task.** Make feed health truthful without disabling the simulated fallback. Implement the optimal solution that: (1) **Distinguishes "live-capable real feed succeeded" from "simulated / no-credential fallback" everywhere status is computed and surfaced** — the `ConnectionTestResult` and the ingestion/feed-health aggregate must classify simulated-only or missing-credential providers as a distinct, non-green state (e.g. `simulated` / `degraded` / `not-live`), never as healthy-green. (2) **Surface an explicit, unmissable indicator** in the dashboard and in the test payloads when market data is simulated or a required credential is missing, including *which* credential env vars are missing — reuse the manager's existing `missing_credential_env_vars` (computed at `services/data_source_manager.py:4895-4952` and `:5810-5836`) rather than recomputing. (3) **Keep the safe/Test profile fully functional** (simulated may still run and populate) but ensure its status can never be rendered as live-feed success; integrate with `PUBLIC_FEED_LIVE_SMOKE` where a real-feed smoke signal is the right gate. Choose the smallest set of changes that makes every health/test path honest, and put the live-vs-simulated decision in one shared helper so no surface can drift back to fake-green.

**Falsify.** With all paid-feed secret stubs empty under the safe/Test profile, prove the health/test surface reports a non-green "simulated / missing-credential" state (not healthy-green) and lists the missing credentials. Then, with a real credential present (or a stubbed live provider in test), prove the same surface reports true live-green. Add a regression test that fails if a simulated-only feed is ever aggregated as healthy-green, and one that fails if the missing-credential indicator is dropped. Do not "fix" it by hiding or removing the simulated source, and do not broaden an `except` to swallow the signal — fix the truthfulness of the status itself.
